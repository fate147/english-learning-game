# CLAUDE.md

减少常见大模型编码错误的行为指南。可根据需要与项目特定说明合并使用。

**权衡：** 这些指南偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考，后编码

**不要臆断。不要隐藏困惑。暴露权衡。**

实现之前：
- 明确说出你的假设。如果不确定，就问。
- 如果存在多种解读，将它们都提出来——不要默默地自行选择。
- 如果存在更简单的方案，就说出来。在必要时提出反对。
- 如果有不清楚的地方，停下来。指出困惑点。提问。

## 2. 简洁优先

**用最少的代码解决问题。不做投机性编写。**

- 不多做未被要求的特性。
- 不为一次性代码做抽象。
- 不做未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行而本可以用 50 行完成，就重写它。

问问自己："高级工程师会觉得这个过于复杂吗？"如果是，就简化。

## 3. 精准改动

**只碰你必须碰的地方。只清理你自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会有不同的做法。
- 如果你发现不相关的死代码，提出来——但不要删除它。

当你的改动产生孤儿代码时：
- 移除**你的改动**导致不再使用的导入/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

检验标准：每一行改动都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到通过。**

将任务转化为可验证的目标：
- "添加校验" → "为无效输入编写测试，然后让测试通过"
- "修复 Bug" → "编写能复现该 Bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后的测试都能通过"

对于多步骤任务，简要说明计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确成功标准可以让你独立迭代。模糊的标准（"让它能跑起来"）需要不断澄清。

---

**这些指南有效的标志是：** diff 中不必要的变更减少了，因过度复杂而导致的返工减少了，澄清性问题发生在实现之前而非犯错之后。
